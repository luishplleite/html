🎯 Visão Geral do Sistema

Sistema: Painel de Validação de Dados - BI Interno para Porto Hack Santos Universitário

Tecnologias: HTML5, CSS3, JavaScript ES6+, Chart.js, Python 3.12

Objetivo: Dashboard para análise de competências, avaliações 360° e validação de dados de participantes

📋 Funcionalidades Principais

  • ✅ Validação de IDs: Verifica consistência entre tabelas relacionadas
  • 📊 Gráficos Radar: Visualização de competências, checkpoints e avaliações
  • 🔍 Filtros Avançados: Filtragem por usuário/time com destaque visual
  • 📈 Análise 360°: Avaliações feitas e recebidas (excluindo mentores)
  • 🏆 Radar de Categorias: Análise por competência específica
  • 👥 Avaliação de Mentores: Feedback de mentores para equipes
  • 🔄 Integração N8N: Webhook para coleta automática de dados

🏗️ Arquitetura Frontend

  • Segregação de Dados: Organização automática por tipo de tabela
  • Renderização Dinâmica: Tabelas e gráficos atualizados em tempo real
  • Estado Reativo: Sincronização entre filtros, seleções e visualizações
  • Cache Inteligente: Backup de dados originais para reset rápido

📊 Descrição Detalhada das Tabelas

👥 USERS

Função: Cadastro de participantes, mentores e staff

  • id - UUID único do usuário (PK)
  • full_name - Nome completo do participante (required)
  • email - Email principal para contato e login (required)
  • role - Papel: member, admin, staff, mentor (required)
  • member_id - ID no Discord (opcional, quando disponível)
🏆 TEAMS

Função: Equipes da maratona

  • id - UUID único da equipe (PK)
  • name - Nome da equipe (required)
  • code - Código de convite da equipe (required)
  • status - CLOSED/OPEN (aceita novos membros?)
  • event_id - FK para events(id) - Evento associado
  • priority - Boolean - Time é prioritário? (opcional)
  • number - Número do time (opcional)
📝 ASSESSMENT_ANSWERS

Função: Respostas das autoavaliações de competências

  • id - ID único da resposta (PK)
  • user_id - FK para users(id) - Quem respondeu
  • team_id - FK para teams(id) - Equipe do respondente
  • question_id - FK para assessment_questions(id) - Competência avaliada
  • answer - Resposta numérica (escala 1-5)
  • moment - INITIAL/FINAL (antes/depois da maratona)
  • submission_id - FK para assessment_submissions(id)
🎯 MENTORINS (F_SUBMISSIONS)

Função: Submissões de avaliações 360° entre participantes

  • id - ID único da submissão (PK)
  • evaluator_id - FK para users(id) - Quem fez a avaliação
  • target_user_id - FK para users(id) - Quem foi avaliado (pode ser null)
  • target_team_id - FK para teams(id) - Equipe avaliada (pode ser null)
  • survey_id - FK para f_surveys(id) - Formulário usado
  • checkpoint_id - FK para checkpoints(id) - Checkpoint da avaliação
  • created_at - Timestamp da avaliação

⚠️ Lógica de Tipos de Avaliação:

  • target_user_id = null, target_team_id ≠ null → Avaliação da equipe inteira
  • evaluator_id = target_user_id, target_team_id = null → Autoavaliação
  • evaluator_id ≠ target_user_id, ambos preenchidos → Avaliação peer-to-peer
❓ ASSESSMENT_QUESTIONS

Função: Questões das 20 competências para autoavaliação

  • id - UUID da questão (PK)
  • text - Texto descritivo da competência (required)
  • category - Categoria da competência (required):
  • - HABILIDADES_BASICAS
  • - HABILIDADES_TECNICAS
  • - COMPETENCIAS_SOCIAIS
  • - COMPETENCIAS_COGNITIVAS
  • - IMPACTO_E_AUTOPERCEPCAO
  • order - Ordem de apresentação no formulário
📊 ASSESSMENT_QUESTIONS_SURVEY

Função: Questões dos formulários de checkpoint (avaliações 360°)

  • id - ID da questão (PK)
  • text - Texto da pergunta (required)
  • type - Tipo de resposta: scale_1_5, text
  • survey_id - FK para f_surveys(id) - Formulário associado

📋 Questões Principais de Liderança (IDs 16-19):

  • ID 16: "Confiança e Clareza"
  • ID 17: "Escuta e Colaboração"
  • ID 18: "Orientação e Ideias"
  • ID 19: "Apoio e Evolução"
📤 F_SUBMISSIONS

Função: Dados derivados de mentorins para compatibilidade

  • submission_id - ID da submissão (copiado de mentorins.id)
  • user_id - FK para users(id) - Quem respondeu
  • question_id - FK para assessment_questions(id) - Questão
  • answer - Resposta numérica (1-5)
  • moment - Momento da resposta (INITIAL/FINAL)
🎮 GUILDS

Função: Servidores Discord associados aos eventos

  • id - ID do Discord (copiado de invite) (PK)
  • event_id - FK para events(id) - Evento associado
  • name - Nome do servidor Discord
  • invite - Código de convite do Discord
  • created_at - Data de criação

🔗 Diagrama de Relacionamento das Tabelas

🔗 Relacionamentos Principais

🎯 Assessment Answers (Tabela Central)

Esta é a tabela central que conecta todas as outras. Contém as respostas das avaliações e autoavaliações.

  • → Users: user_id identifica quem respondeu
  • → Teams: team_id identifica a equipe do respondente
  • → Assessment Questions: question_id identifica qual competência foi avaliada
  • → F Submissions: submission_id agrupa respostas por submissão
🤝 Mentorins (Avaliações 360°)

Registra todas as submissões de avaliações entre participantes, incluindo avaliações de mentores.

  • evaluator_id → Users - Quem fez a avaliação
  • target_user_id → Users - Quem foi avaliado (pode ser nulo para avaliações de equipe)
  • target_team_id → Teams - Equipe avaliada (pode ser nulo para avaliações pessoais)
🏆 Fluxo de Dados para Gráficos
  1. Competências 21D: Assessment_Answers → Assessment_Questions (por category)
  2. Avaliações 360°: Mentorins → Users (evaluator/target) → F_Answers (respostas)
  3. Checkpoint: Mentorins (por checkpoint_id) → F_Answers (questões 16-19)
  4. Mentores: Mentorins + Users (role='mentor') → F_Answers

🔌 APIs e Endpoints

📡 Fonte de Dados Principal

const DATA_URL = 'https://n8n.hackathon.souamigu.org.br/webhook/dadosportohacker';

Método: GET

Formato: JSON Array com dados segregados automaticamente

🛠️ Endpoints do Servidor Python

  • GET / - Página principal (index.html)
  • GET /api/supabase-url - URL do Supabase
  • GET /api/supabase-key - Chave pública do Supabase
  • GET /api/supabase-config - Configuração completa do Supabase
  • POST /api/configure-n8n-webhook - Configurar webhook N8N
  • GET /webhook/n8n-teams-data - Testar webhook N8N

🔄 Fluxo de Processamento de Dados

  1. Carregamento: Fetch dos dados do webhook N8N
  2. Segregação: Organização por tipo de tabela usando segregateData()
  3. Cache: Backup em originalSegregatedData
  4. Renderização: Criação de tabelas e gráficos
  5. Interatividade: Filtros e seleções em tempo real

⚙️ Funções Principais para Frontend

  • segregateData(data) - Organiza dados brutos por tabela
  • renderAllTables(data, order) - Renderiza todas as tabelas
  • renderCharts(data) - Cria todos os gráficos radar
  • validateRelatedIds() - Valida consistência entre IDs
  • applyAdvancedFilter(selection) - Aplica filtros avançados
  • clearAdvancedFilter() - Reset para dados originais

📊 Estrutura de Retorno do Webhook

O webhook retorna um array com objetos que são automaticamente classificados por:

  • Teams: item.code && item.name && item.status
  • Users: item.full_name && item.role
  • Mentorins: item.evaluator_id
  • Assessment Answers: item.question_id && item.answer !== undefined
  • Assessment Questions: item.text && item.category
  • Survey Questions: item.text && item.type && item.survey_id
  • Guilds: item.name && item.invite
